# v0.3.35 — one-command fresh-user reset (retro)

**Released**: 2026-04-28 · **Commit**: [`3ceccb7`](../../CHANGELOG.md#0335---2026-04-28) · **Tag**: `v0.3.35`

## Background

The previous `paperwiki uninstall` only handled the plugin layer
(roughly 30% of paper-wiki's on-disk footprint). Anyone testing
fresh-install behaviour — or switching machines — still had to follow
up with a 5-command manual sequence: `paperwiki uninstall` plus four
`rm -rf` invocations covering `~/.config/paper-wiki/`, the
`~/.local/bin/paperwiki` shim, the marketplace clone, and a Python
edit of the `~/.claude/settings.json` `extraKnownMarketplaces` block.
The mismatch between the verb "uninstall" (implies "remove every
trace") and the actual semantics (plugin layer only) was the
proximate reason every fresh-user simulation took longer than it
should have. v0.3.35 collapses the manual ritual into a single
flag-driven invocation.

## What shipped

- `paperwiki uninstall` — unchanged plugin-layer-only default. Safe
  by design; no surprise destruction.
- `paperwiki uninstall --everything --yes` — adds
  `~/.config/paper-wiki/` (recipes + secrets + venv), the
  `~/.local/bin/paperwiki` shim and its `.paperwiki-path-warned`
  marker, the marketplace clone, and the
  `settings.json::extraKnownMarketplaces.paper-wiki` entry.
- `paperwiki uninstall --everything --purge-vault PATH --yes` —
  adds **surgical** removal of paperwiki-created vault content
  (`Daily/`, `Wiki/`, `.digest-archive/`, `.vault.lock`,
  `Welcome.md`) while preserving `.obsidian/`, `.DS_Store`, and any
  unrelated user content.
- `paperwiki uninstall --everything --purge-vault PATH --nuke-vault --yes`
  — upgrades the surgical removal to `rm -rf PATH` for a clean-slate
  vault wipe.
- Idempotent re-runs: a second invocation on a clean state exits 0
  with `nothing to remove` rather than erroring on missing targets.

Internally, orchestration moved from inline helpers in `cli.py` to a
new `paperwiki.runners.uninstall` module so tests can drive the
orchestrator directly via `tmp_path` fixtures.

## Lessons learned

The previous `uninstall` semantics (plugin-layer only) were technically
honest — the plugin layer is what `claude plugin uninstall` removes —
but they were **incomplete** relative to user expectation. The verb
implied "remove every trace", not "remove the part installed by the
plugin manager". v0.3.35 keeps the safe-by-default contract (no flag
= no surprise destruction) while making the full reset a one-liner.
The flag-driven design beats a separate `paperwiki nuke` subcommand
because flags compose: a user can opt into the user-config wipe
without touching a vault, or vice versa, without learning a second
command. Vault handling stays its own opt-in flag (`--purge-vault
PATH`) rather than a default behaviour — the user must point at the
vault explicitly to opt into vault changes.

The fresh-user simulation enabled by this release immediately paid
for itself: the next run (which led to v0.3.36) caught a privacy
regression in the setup wizard — Claude was prompting for the S2 API
key inline via `AskUserQuestion`, leaking the key into the session
transcript. v0.3.36 closed that hole. See [v0.3.36 in
CHANGELOG.md](../../CHANGELOG.md#0336---2026-04-28).
